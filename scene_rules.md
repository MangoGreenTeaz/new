# `scene_label.py` 场景规则维护说明

本文档用于维护 `scene_label.py` 中场景标注规则的人类可读说明。

代码中的场景规则以优先级顺序依次执行：

1. 先判断高优先级场景
2. 仅对当前 `scene_label` 仍为空字符串的记录继续判断下一个场景
3. 一旦某条记录被命中并赋值，后续场景不再覆盖该记录

## 当前代码状态

- 脚本文件：`scene_label.py`
- 输入文件默认值：`out.csv`
- 输出文件默认值：`scene_out.csv`
- 当前输出新增列：`scene_label`
- 当前规则状态：已实现 `high_speed_rail`、`airport`、`self_drive`、`service_area_rest`、`fuel_charge_stop`、`parking_pickup_dropoff`、`tourism`、`hotel_stay`、`cultural_venue_visit`、`shopping`、`outdoor_sports`、`family_fun`、`ride_hailing_passenger`、`worker_active`、`worker_rest`、`subway_travel`、`commuting_to_work`、`commuting_home`、`trip_planning` 十九个场景

## 规则维护约定

后续每新增或修改一个场景时，需要同步修改两个地方：

1. `scene_label.py`
   - 在 `build_scene_rules()` 中新增或修改规则
   - 指定场景名称、优先级、标签值、说明和匹配条件
2. `scene_rules.md`
   - 更新下方规则表
   - 确保文档描述与代码逻辑一致

## 实现原则

- `scene_label` 默认值为空字符串 `""`
- 所有场景按优先级从小到大执行
- 后续场景只能处理尚未标注的记录
- 场景扫描过程中一旦遇到 `scene_label` 已非空的记录，会立刻截断当前候选范围，不跨越已标注数据继续扩展
- 数据按 `udid`、`time` 排序后处理，场景判断严格限制在同一个用户内
- 分块处理时会缓存每个块尾部最后一个 `udid` 的数据，等待和下一块拼接后再统一判断，避免同一用户被切断
- 若新增场景依赖新的输入字段，需要同步更新输入字段说明文档
- 若调整场景优先级，需要同步更新本文件中的规则表

## 已实现场景说明

### 高铁场景

#### 场景范围识别

1. 在同一个 `udid` 内按时间顺序扫描
2. 遇到 `poi` 包含 `高铁站` 的首条记录，作为候选范围起点，并纳入范围
3. 从起点继续向后扫描，只要持续出现“高铁相关特征”，就继续扩展范围
4. 若连续 3 条记录都不含高铁相关特征，则停止扫描
5. 候选范围终点取“最后一条仍然具有高铁相关特征的记录”
6. 若扫描过程中遇到已被更高优先级场景标注的记录，则立即停止当前高铁候选范围扩展

#### 高铁相关特征

满足以下任一条件即视为高铁相关特征：

- `poi` 包含 `高铁站`
- `app_ticket == true`
- `move_fast == true`
- `move_cross_city == true`

#### 场景丢弃条件

若候选范围满足以下任一条件，则整个高铁场景范围丢弃，不进行标注：

- 范围内任意记录的 `poi` 包含 `机场`
- 范围内没有任何记录满足 `move_cross_city == true`
- 范围内不存在节点 B 或节点 C

#### 节点定义

- 节点 A：候选范围第一条记录，标注为 `抵达始发高铁站`
- 节点 B：候选范围内第一次出现 `move_fast == true` 的记录，标注为 `高铁行程途中`
- 节点 C：
  - 先取候选范围最后一条记录的 `city` 作为“最后一个城市”
  - 再从范围起点向后找第一次出现该城市的记录
  - 若该记录不在节点 B 之后，则继续向后找第一个位于节点 B 之后、且城市等于最后一个城市的记录
  - 该记录标注为 `抵达终点高铁站`

#### 区间标注规则

- 节点 A：`抵达始发高铁站`
- A 和 B 之间：`高铁站候车`
- 节点 B：`高铁行程途中`
- B 和 C 之间：`高铁行程途中`
- 节点 C：`抵达终点高铁站`
- C 之后直到场景范围最后一条：`离开终点高铁站`

### 自驾场景

#### 场景范围识别

1. 在同一个 `udid` 内按时间顺序扫描
2. 同时满足 `move_fast == true` 且 `move_cross_city == true` 的记录作为自驾锚点
3. 从锚点向前、向后分别扫描，只要持续出现“自驾相关特征”，就继续扩展范围
4. 若某个方向连续 3 条记录都不含自驾相关特征，则该方向停止扫描
5. 范围起点和终点分别取该方向上最前/最后一条仍具有自驾相关特征的记录
6. 若任一方向扫描过程中遇到已被更高优先级场景标注的记录，则该方向立即停止扩展

#### 自驾相关特征

满足以下任一条件即视为自驾相关特征：

- `move_fast == true`
- `move_cross_city == true`
- `poi` 包含 `加油站`
- `poi` 包含 `电动车充电站`
- `poi` 包含 `服务区`
- `poi` 包含 `停车场`

#### 场景丢弃条件

若候选范围满足以下任一条件，则整个自驾场景范围丢弃，不进行标注：

- 范围内不存在任何 `move_cross_city == true` 的记录
- 范围内任意记录的 `poi` 包含 `高铁站`
- 范围内任意记录的 `poi` 包含 `机场`

#### 范围内标签优先级

对自驾范围内的每条记录按以下优先级依次判断：

1. 包含 `服务区` POI：`服务区休息`
2. 包含 `加油站` 或 `电动车充电站` POI：`驾驶途中加油、充电`
3. 包含 `停车场` POI：`去停车场停车/取车`
4. 其余记录：`自驾途中`

#### 范围最后一条特殊规则

- 若范围最后一条包含 `停车场` POI，则标记为 `去停车场停车/取车`
- 否则标记为 `驾车抵达终点`

### 旅游

#### 锚点条件

- 当前记录未标注
- 当前记录的 `poi` 包含 `旅游景点`
- 当前记录满足 `app_travel == true`
- 当前记录不是 `凌晨`，即 `time_early_morning != true`

#### 密度校验

- 从锚点开始向后最多扫描 10 条后续记录（包含锚点在内共最多 11 条）
- 若途中遇到 `凌晨`，立即停止该窗口扫描
- 要求窗口内至少有 2 条记录满足 `poi` 包含 `旅游景点`（包含锚点本身）
- 若不足 2 条，则该锚点视为噪点，不进行旅游场景标注

#### 结束边界

- 从锚点继续向后扫描
- 若扫描过程中遇到已标注记录，立即停止当前旅游候选范围扩展
- 若遇到 `凌晨`，立即停止
- 若连续 5 条记录同时不包含 `旅游景点` POI 且不满足 `app_travel == true`，立即停止
- 最终边界取最后一次出现 `旅游景点` POI 或 `app_travel == true` 的记录位置 `last_spot_idx`

#### 实际打标

仅在 `[锚点, last_spot_idx]` 范围内遍历，并且只对当前仍未标注、且 `poi` 包含 `旅游景点` 的记录进行标注：

- 满足 `move_any == true`：标记为 `旅游参观`
- 满足 `move_any == false`：标记为 `旅游中途休息`

### 酒店办理入住、旅游住宿休息

#### 范围识别逻辑

- 在同一用户内按时间顺序扫描
- 第一次扫描到 `poi` 包含 `酒店旅馆` 的记录，作为锚点，也是场景范围第一条
- 从锚点继续向后扫描
- 若连续 30 条记录都不包含 `酒店旅馆` POI，则停止扫描
- 场景范围最后一条取“最后一条仍包含 `酒店旅馆` POI 的记录”

#### 标注规则

### 网约车乘客出行场景

#### 范围与节点判断逻辑

- 从前往后寻找候选起点 `A`
- 起点 `A` 需要同时满足：
  - 当前索引不小于当前扫描位置
  - `app_ride_hailing == true`
  - 当前记录未标注
- 若 `A` 本身已经处于移动状态，即 `move_any == true`，则跳过该候选
- 找到 `A` 后，继续向后找到第一个已被其他场景标注的位置 `next_labeled_idx`
- 后续所有搜索不能越过该已标注边界
- 在 `A` 后最多看 5 条记录，且不能越过 `next_labeled_idx - 1`
- 若这个窗口内没有任何移动记录，则放弃本次候选
- 从 `A` 到边界之间找到第一条移动记录 `first_move`
- 再从 `first_move` 开始向后找到连续移动段的最后一条 `last_move`
- 若 `first_move == A`，说明没有“等待后上车”的阶段，不符合规则，跳过该候选

#### 小场景标注逻辑

- `A` 到 `first_move - 1`：标记为 `等待网约车`
- `first_move` 到 `last_move - 1`：标记为 `乘坐网约车行程中`
- `last_move`：标记为 `网约车到达终点`
- 标注完成后，从 `last_move + 1` 继续寻找下一个片段

### 地铁出行场景

#### 范围与节点判断逻辑

- 从前往后寻找候选起点 `A`
- 起点 `A` 需要同时满足：
  - `poi` 包含 `地铁站`
  - 当前记录没有任何移动，即 `move_any == false`
  - 当前记录未标注
- 找到 `A` 后，向后找到下一个已标注位置 `next_labeled_idx`，作为本段上界
- 在 `A` 后最多看 5 条记录，若该窗口内没有任何 `move_fast == true` 的记录，则认为不是地铁场景

#### 终止点识别

- 从 `A + 1` 往后扫描
- 若发现连续两条都 `move_any == false`，则认为行程已结束，并记第一条非移动位置为 `stop_idx_first_nonmove`
- 若未找到该停止信号，则把 `A` 到 `next_labeled_idx` 内最后一条移动记录作为 `last_move`
- 若找到了停止信号，则从 `stop_idx_first_nonmove - 1` 向前回找最近一条移动记录作为 `last_move`

#### 终点站识别

- 在 `A` 到 `last_move` 范围内，找最后一个仍未标注且 `poi` 包含 `地铁站` 的位置作为 `end_poi`
- 若 `end_poi == A`，或 `A+1:last_move` 之间完全没有移动记录，则该段不成立

#### 小场景标注逻辑

- `A`：标为 `抵达始发地铁站`
- `end_poi`：标为 `抵达终点地铁站`
- `A+1` 到 `last_move`：
  - 跳过 `end_poi`
  - 只对“未标注且有移动”的记录标为 `乘坐地铁中`
