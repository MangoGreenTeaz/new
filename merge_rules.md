## 1. 整体流程概述

系统通过流式分块的方式读取 CSV 原始文件，按顺序遍历每条用户的交互记录。在这个过程中，为每个用户（`udid`）维护一个固定大小的**历史文本窗口**。每读入一条记录，就会将**当前的请求文本**、**关联的订单信息**、**距离出行/抵达的动态时间提示**以及**该用户最近几次的历史请求**按照特定格式拼接成一个完整的字符串（`MERGED_TEXT`）。

默认输入输出链路：`data_feature_label_order.csv` → `data_feature_label_order_merge.csv`

**数据前提假设：**
1. 数据已经按照 `udid`（用户设备ID）和 `time`（交互时间）排列好。
2. 同一个 `udid` 的记录是连续出现的。

## 2. 涉及的数据字段

程序在读取和写出时主要处理以下字段。其中，用于生成 `MERGED_TEXT` 的核心字段加粗显示：

| 字段名称 | 描述 | 在拼接中的作用 |
| :--- | :--- | :--- |
| **`time`** | 当前交互的发生时间 (格式：`YYYY/MM/DD HH:MM`) | 用于与订单中的发车/起飞时间做对比计算时间差 |
| **`udid`** | 用户设备唯一标识 | 用于隔离不同用户的历史记录窗口（切换 `udid` 时清空历史） |
| `scene_label` | 场景标签 | 透传字段（输入中为 `label`，输出统一重命名为 `scene_label`） |
| **`text`** | 当前用户交互的文本内容 | 拼接到 `[current]` 标签内 |
| **`order`** | 当前交互关联的订单内容 | 拼接到 `[order]` 标签内，并解析其中的关键时间 |
| `context` | 上下文环境信息 | 透传字段，不参与文本合并 |
| `history_usage` | 历史服务使用情况 | 透传字段，不参与文本合并 |
| `service_click` | 服务点击信息 | 透传字段，不参与文本合并 |

## 3. 文本拼接核心逻辑

最终生成的 `MERGED_TEXT` 包含三个主要部分，按顺序拼接：**当前文本 + 订单与时间提示 + 历史记录**。

### 3.1 基础请求（Current）

当前的请求文本被包裹在 `[current]...[/current]` 标签中。这是所有拼接的基础。
*   **格式**：`[current]{当前text}[/current]`

### 3.2 订单及时间提示（Order & Time Hints）

如果在当前记录中，存在 `order`（订单）数据，它会被**包含在 `[current]` 内部，紧跟在 `{当前text}` 之后**。

同时，如果该订单文本包含了特定的时间信息（如火车票、机票），系统会计算**当前时间**与**订单时间**的差值，并动态生成提示文案。

**1. 订单拼接格式：**
*   `[order]{当前order内容}[/order]`

**2. 时间提示触发条件：**
*   订单文本中包含正则表达式匹配的字段：`出发时间：YYYY/MM/DD HH:MM` 或 `到达时间：YYYY/MM/DD HH:MM`。

**3. 时间窗口计算规则：**
*   计算当前 `time` 与解析出的订单时间（出发/到达）的绝对时间差（分钟）。
*   若**任一时间差 <= `time_window_minutes`（默认 120 分钟）**，或者配置参数为 `0`（强制所有订单显示），则触发文案生成。
*   一旦触发，出发和抵达的时间差都会被显示。

**4. 提示文案格式：**
*   紧跟在 `[/order]` 之后，格式如：`距出发时间+xx分钟，距抵达时间-yy分钟`。
*   **符号含义**：
    *   `+`：表示该时间点**尚未到来**（或刚刚好到达为 `+0`），例如：距起飞还有 30 分钟。
    *   `-`：表示该时间点**已经过去**，例如：飞机已经起飞 10 分钟。

### 3.3 历史记录（Previous）

在当前记录的闭合标签 `[/current]` 之后，会追加用户之前的历史请求文本。

**1. 历史窗口机制：**
*   系统维护一个最大长度为 `n`（默认 10）的队列，仅保存历史记录中的 `text` 字段。
*   当读取到一条新记录时，如果发现其 `udid` 与上一条不同，队列会被清空，避免串用他人历史。

**2. 倒序拼接规则：**
*   距离当前越近的历史记录，编号越小。
*   **格式**：` [previous-1]{上一条历史文本}[/previous-1] [previous-2]{更早一条历史文本}[/previous-2] ...`

## 4. 配置参数说明

脚本底部的 `config` 字典控制了整个合并流程的行为：

| 参数 | 类型 | 默认值 | 作用说明 |
| :--- | :--- | :--- | :--- |
| `input_file` | string | `data_feature_label_order.csv` | 输入数据的 CSV 文件路径 |
| `output_file` | string | `data_feature_label_order_merge.csv` | 拼接完成后输出的 CSV 文件路径 |
| `n` | int | `10` | 历史记录窗口的大小，决定向前追溯几条历史交互 |
| `batch_size` | int | `200000` | 流式处理时每次读取和写入内存的数据行数，用于控制内存占用 |
| `time_window_minutes` | int | `120` | 时间提示触发窗口（分钟）。订单出发/抵达时间与当前时间差在此范围内时生成文案。**若设为 `0`，则对含有出发/到达时间的订单强制每条生成提示**。 |

## 5. 最终输出样例

假设参数 `n=2`, `time_window_minutes=120`。

**场景：用户正在去机场的路上，并在途中发起了多次交互。**

*   **第 1 条请求**（无历史，仅文本）：
    *   **text**: "打开健康码"
    *   **MERGED_TEXT**: `[current]打开健康码[/current]`
*   **第 2 条请求**（有 1 条历史，无订单）：
    *   **text**: "帮我查一下去机场路况"
    *   **MERGED_TEXT**: `[current]帮我查一下去机场路况[/current] [previous-1]打开健康码[/previous-1]`
*   **第 3 条请求**（有 2 条历史，触发订单与时间提示。假设起飞还剩 30 分钟，抵达还剩 180 分钟）：
    *   **text**: "我的登机口在哪"
    *   **order**: "国航CA1234 出发时间：2024/05/20 14:00 到达时间：2024/05/20 16:30"
    *   **MERGED_TEXT**: `[current]我的登机口在哪[order]国航CA1234 出发时间：2024/05/20 14:00 到达时间：2024/05/20 16:30[/order]距出发时间+30分钟，距抵达时间+180分钟[/current] [previous-1]帮我查一下去机场路况[/previous-1] [previous-2]打开健康码[/previous-2]`
